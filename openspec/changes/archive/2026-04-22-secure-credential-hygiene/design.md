## 背景

当前项目已经有可运行的多账户 MT5 -> Notion 同步链路，但它的安全边界仍然是“约定式”的，而不是“默认安全”的。也就是说，团队知道 `.env` 和 `accounts.json` 不该提交，但仓库中仍然存在会误导或泄露敏感信息的内容：示例文件中混入了真实账户；历史归档脚本带着生产标识；运行日志和诊断输出回显了过多账号和环境细节。

这类问题的危险不在于单次运行，而在于复制、分享、备份和协作。一旦某个文件“看起来像示例、实际上是真数据”，或者某个诊断脚本“看起来只是测试、实际上会打印敏感标识”，项目就会在最普通的日常操作里泄露信息。

本次设计追求的是基础安全卫生，而不是企业级密钥托管。目标是把仓库与运行输出收敛到“默认最小暴露”，让开发、排查和分享时不需要依赖操作者自己记得处处避坑。

## 目标 / 非目标

**目标：**
- 让所有允许进入仓库的示例文件只包含占位符或假数据，不再携带真实凭据。
- 让主同步脚本和诊断工具默认不输出完整 MT5 登录号、余额、终端数据路径、Notion token 前缀或完整数据库 ID。
- 清理 `archive/` 目录中仍可能传播生产标识或误导使用者的历史残留。
- 在文档中明确区分“本地私有配置”和“可共享示例文件”，并提醒历史日志的分享风险。
- 为新的脱敏/净化行为提供基本回归保护。

**非目标：**
- 不改变 MT5 -> Notion 的同步业务语义。
- 不引入 Windows Credential Manager、DPAPI、外部 Secret Manager 或加密配置文件。
- 不自动删除用户本地已有的 `.env`、`accounts.json` 或历史日志文件。
- 不试图恢复或追踪已经离开当前工作区的敏感信息传播面。

## 设计决策

### 1. 共享文件与私有配置必须被明确分层

仓库中允许存在的配置文件只能是：
- `.env.example`
- `accounts.example.json`

这两类文件都只能承载占位符、假数据或说明性内容，不能包含任何可直接登录或访问生产资源的值。真正的运行配置继续放在被 `.gitignore` 忽略的本地文件里，例如 `.env` 和 `accounts.json`。

这个决策的重点不是“再提醒一次不要提交”，而是让仓库本身不再携带会让人误判的文件。只要文件名里有 `example`，它就必须真的是示例。

### 2. 运行日志采用“最小可诊断信息”策略

主脚本的日志仍然需要保留这些诊断维度：
- 当前处理的是哪个逻辑账户
- 当前步骤是初始化、登录、拉取、同步还是重试
- 每个账户的 created / updated / duplicate / failed 汇总

但它不再默认输出这些原始值：
- 完整 MT5 登录号
- 账户余额
- 服务器名
- MT5 终端数据路径
- Notion token 或数据库标识

为此，日志层应优先使用 `account_name` 作为可观察身份，而不是 `login/server` 的原始组合。换句话说，账户可追踪，但底层凭据与环境细节不应成为默认日志内容。

### 3. 诊断工具只报告“是否已配置 / 是否可连接”，不回显敏感值

诊断工具的价值在于回答：
- 配置是否存在
- 客户端是否能初始化
- 网络或接口是否连通

而不是把 token、数据库 ID 或账户值打印给操作者确认。因此，输出应从“值回显”转成“状态回显”，例如：
- `NOTION_TOKEN configured`
- `DATABASE_ID configured`
- `MT5 credentials configured`

如果确实需要更强的可观测性，也应采用掩码形式，而不是完整值或前缀回显。

### 4. `archive/` 目录按“历史参考”而不是“备用入口”处理

`archive/` 中的脚本不是当前受支持的主路径，但只要它们还留在仓库里，就仍然会被搜索、复制和误运行。因此有两个必要动作：
- 去掉其中的硬编码生产标识与明显的敏感残留
- 在文件头或文档层面明确它们是历史参考，而不是当前推荐入口

这不是为了维护归档脚本的功能一致性，而是为了阻断它们继续成为敏感信息的传播载体。

### 5. 用轻量回归保护“脱敏不会回退”

这次改动的风险不是功能坏掉，而是以后又有人顺手把账号、token 前缀或真实示例数据加回去。因此测试重点不应放在端到端联网，而应放在这些纯逻辑约束：
- 脱敏辅助函数是否按预期返回
- 敏感日志字段是否不再出现在关键输出中
- 示例账户文件是否保持占位符结构

这样可以用低成本的单元测试保护高风险行为边界。

## 风险 / 权衡

- 日志会比以前“少信息”，部分排查习惯需要切换到更抽象的账户级视角。
- 清理 `archive/` 中的硬编码值会让它们不再是完全原样的历史快照，但这是合理取舍；当前更重要的是停止传播敏感信息。
- 这次设计无法解决“凭据是否应该长期保存在本地明文文件”这个更深层问题，它只是先把当前最明显的暴露面收住。
- 如果真实凭据已经外泄，代码收口不能替代密码轮换；两者是不同层面的动作。

## 迁移计划

1. 先把仓库中的共享文件净化为示例状态。
2. 再收紧运行日志和诊断输出，确保新的运行不会继续制造敏感残留。
3. 更新 README，明确本地私有配置的放置方式，并提醒旧日志可能包含历史敏感信息。
4. 最后补充回归测试，防止未来回退。

## 待确认问题

- 是否希望继续保留 `archive/` 目录为“可执行历史脚本”，还是只把它当作参考文本保留。
- 是否希望在后续 change 中继续推进到“本机受保护凭据存储”，例如 Windows Credential Manager。
