## 背景

当前项目是一个单文件为主的 Python 同步脚本，外围辅以若干手工诊断工具。主流程已经能完成 MT5 到 Notion 的同步，但工程边界薄弱：敏感配置和项目路径没有统一入口，不同脚本各自处理连接逻辑和错误输出，导致安全问题、统计失真和运行环境敏感问题反复出现。

本次变更不引入新的外部服务，也不改变 MT5 或 Notion 的核心集成方式，而是在现有脚本结构上收敛风险并建立最基本的可验证护栏。

## 目标 / 非目标

**目标：**
- 消除仓库中的明文 MT5 凭据，并统一敏感配置读取方式。
- 让主同步流程能够明确区分创建、更新、重复跳过和失败，避免误报。
- 让主脚本和诊断工具使用仓库相对路径解析 `.env`、映射文件和日志目录。
- 为关键纯逻辑补充自动化测试，降低回归风险。

**非目标：**
- 不重写为多模块架构，也不引入复杂的包结构。
- 不改变 Notion 数据模型，不新增新的业务字段。
- 不处理 MT5 成交匹配算法以外的更深层交易语义建模问题。
- 不覆盖所有历史 `archive/` 脚本，只收敛当前仍可能被误用的诊断入口。

## 设计决策

1. 引入共享的项目路径与环境加载辅助函数。
   主脚本使用 `Path(__file__).resolve().parent` 作为项目根，显式加载根目录 `.env`，并将日志目录和映射文件路径都解析到根目录下。这样计划任务、IDE 或任意当前工作目录都不会改变行为。

2. 将单笔同步结果建模为显式状态。
   `sync_trade` 不再只返回布尔值，而是返回有限状态值，例如 `created`、`updated`、`duplicate`、`failed`。批量同步函数根据状态累积统计并输出精确摘要；只要存在失败，进程就应返回非零退出码。

3. 保持主流程最小改动，优先提炼纯函数。
   对 Notion payload 组装、路径解析和批量统计等逻辑提炼出可测试的纯函数或小方法，在不重构整体结构的前提下，让测试可以直接覆盖关键行为。

4. 诊断工具统一改为环境变量登录。
   现有工具中的硬编码 `mt5.initialize(login=..., password=..., server=...)` 改为复用环境变量。若缺少配置，则输出明确提示并退出，而不是继续用危险默认值。

5. 使用标准库 `unittest` 建立基础测试。
   当前项目没有现成测试框架或依赖清单。为了避免扩大依赖面，本次新增 `tests/` 并使用 `python -m unittest` 运行，覆盖最关键的纯逻辑和统计逻辑。

## 风险 / 权衡

- 仍然保留单文件主脚本，意味着结构复杂度不会一次性下降很多，但可以用最小风险完成当前收口。
- 将失败显式暴露后，历史上“看似成功”的运行可能变成失败退出；这是预期行为变化，需要在 README 中说明。
- 部分诊断脚本可能因为缺少环境变量而无法直接运行，但这是安全换取的必要约束。

## 迁移计划

1. 创建 `.env.example`，明确 MT5 和 Notion 所需配置项。
2. 更新主脚本为项目根路径解析和显式状态统计。
3. 清理诊断脚本中的硬编码凭据，统一改为环境变量加载。
4. 增加 `tests/` 与依赖清单，并执行自动化测试。
5. 更新 README，说明新的运行方式、测试命令和失败统计语义。

## 待确认问题

- 现有诊断脚本是否全部需要长期保留；本次先修正明显危险和失效的脚本，后续可考虑再进一步归并。
- 是否需要在后续 change 中把主脚本拆分为模块化包结构；本次不做该级别重构。
